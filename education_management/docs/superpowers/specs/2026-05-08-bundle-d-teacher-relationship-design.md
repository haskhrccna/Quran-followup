# Bundle D: Teacher Relationship Features — Design Spec
**Date:** 2026-05-08
**Scope:** Backend (new model + 3 endpoints) + Mobile (3 modified screens + 2 new screens).

---

## Goal

Let students see who their assigned teacher is, and let them request a teacher change with a reason. Admins process these requests. Teachers can see which of their students have requested a change.

---

## Architecture

The student-teacher relationship already exists via `Appointment` (status=`ACCEPTED`). The `getMyAppointments` endpoint already returns `teacher.firstName/lastName/email` for students — **no new endpoint needed for teacher name display**.

What's new: a `TeacherChangeRequest` model with submit / list / decide endpoints.

---

## Database Changes

**New enum** (add to `schema.prisma`):
```prisma
enum TeacherChangeStatus {
  PENDING
  APPROVED
  DENIED
}
```

**New model** (add to `schema.prisma`):
```prisma
model TeacherChangeRequest {
  id               String              @id @default(cuid())
  studentId        String
  currentTeacherId String
  reason           String
  status           TeacherChangeStatus @default(PENDING)
  adminNote        String?
  createdAt        DateTime            @default(now())
  updatedAt        DateTime            @updatedAt

  student        User @relation("StudentChangeRequests", fields: [studentId], references: [id], onDelete: Cascade)
  currentTeacher User @relation("TeacherChangeRequests", fields: [currentTeacherId], references: [id], onDelete: Cascade)

  @@index([studentId])
  @@index([currentTeacherId])
  @@index([status])
}
```

**User model additions** (add two new relation fields):
```prisma
teacherChangeRequestsAsStudent  TeacherChangeRequest[] @relation("StudentChangeRequests")
teacherChangeRequestsAsTeacher  TeacherChangeRequest[] @relation("TeacherChangeRequests")
```

## API Endpoints

### POST `/api/v1/teacher-changes`
**Auth:** STUDENT only
**Body:** `{ reason: string }` (min 10 chars, max 500 chars)
**Logic:**
1. Find student's ACCEPTED appointment → get `currentTeacherId`
2. If no accepted appointment → 400 "You have no assigned teacher"
3. If student already has a PENDING request → 409 "You already have a pending request"
4. Create `TeacherChangeRequest` with `studentId`, `currentTeacherId`, `reason`, `status: PENDING`
5. Return created request

### GET `/api/v1/teacher-changes`
**Auth:** Any authenticated user
**Logic by role:**
- `STUDENT`: returns their own requests (all statuses), includes `currentTeacher.firstName/lastName`
- `TEACHER`: returns PENDING requests where `currentTeacherId === userId`, includes `student.firstName/lastName/email`
- `ADMIN`: returns all requests (filter by `?status=PENDING` when provided), includes both `student` and `currentTeacher` names

### PATCH `/api/v1/teacher-changes/:id`
**Auth:** ADMIN only
**Body:** `{ action: 'APPROVE' | 'DENY', adminNote?: string }`
**Logic:**
1. Find request by id — 404 if not found
2. If status !== PENDING → 409 "Request already decided"
3. Update status to APPROVED or DENIED, set `adminNote`
4. Return updated request

## Mobile API Layer

**File:** `mobile/src/api/teacherChange.ts`

```typescript
import { apiClient } from './client';

export const teacherChangeApi = {
  submit: async (reason: string) => {
    const res = await apiClient.post('/teacher-changes', { reason });
    return res.data;
  },
  list: async (status?: 'PENDING') => {
    const res = await apiClient.get('/teacher-changes', { params: status ? { status } : {} });
    return res.data;
  },
  decide: async (id: string, action: 'APPROVE' | 'DENY', adminNote?: string) => {
    const res = await apiClient.patch(`/teacher-changes/${id}`, { action, adminNote });
    return res.data;
  },
};
```

## New Hook

**File:** `mobile/src/hooks/useTeacherChange.ts`

```typescript
import { useState, useCallback } from 'react';
import { teacherChangeApi } from '../api/teacherChange';

export function useTeacherChange() {
  const [requests, setRequests] = useState<any[]>([]);
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const fetchRequests = useCallback(async (status?: 'PENDING') => {
    setIsLoading(true);
    setError(null);
    try {
      const data = await teacherChangeApi.list(status);
      setRequests(data);
    } catch {
      setError('Failed to load requests');
    } finally {
      setIsLoading(false);
    }
  }, []);

  const submitRequest = useCallback(async (reason: string) => {
    await teacherChangeApi.submit(reason);
  }, []);

  const decideRequest = useCallback(async (id: string, action: 'APPROVE' | 'DENY', adminNote?: string) => {
    await teacherChangeApi.decide(id, action, adminNote);
    await fetchRequests();
  }, [fetchRequests]);

  return { requests, isLoading, error, fetchRequests, submitRequest, decideRequest };
}
```

## New Screen — Student Teacher Change Request

**File:** `mobile/app/student/teacher-change.tsx`

- Header: back arrow + title `t('requestTeacherChange')`
- On mount: fetch existing requests via `useTeacherChange().fetchRequests()`
- **If PENDING request exists:** show a status card ("Your request is under review") — no form
- **If APPROVED or DENIED:** show decision status + `adminNote` — no form
- **If no request:** show the submission form:
  - Info text showing current teacher name (from route param or re-fetched appointments)
  - Multiline `TextInput` for reason (shows character count, min 10)
  - Submit button (disabled when `reason.trim().length < 10` or submitting)
  - On success: Alert `t('requestSubmitted')` → `router.back()`
  - On 400/409 error: show inline error text below the form

## New Screen — Admin Change Requests

**File:** `mobile/app/admin/change-requests.tsx`

- Header: back arrow + title `t('teacherChangeRequests')`
- On mount: `fetchRequests()` with no filter (returns all)
- Toggle chips: All / Pending / Approved / Denied
- `FlatList` of requests
- Each row: student name, current teacher name, reason (1 line truncated), status badge, date
- Tapping a row expands it inline to show: full reason, admin note input, Approve button, Deny button
- Approve/Deny: calls `decideRequest(id, action, adminNote)` → list re-fetches → row collapses
- Empty state: `t('noChangeRequests')`

Add a "Change Requests" button in admin home header (or accessible from the existing settings/menu area).

## Error Handling

- Student submits with no ACCEPTED appointment → server 400 → show inline "No teacher assigned to you yet"
- Student submits duplicate PENDING → server 409 → show inline "You already have a pending request"
- Admin decides already-decided request → server 409 → re-fetch list (stale UI)
- All network errors → inline error text + retry (no crashes)
