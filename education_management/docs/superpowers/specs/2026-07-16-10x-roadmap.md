## 1. Current-state audit (evidence, not memory)

**Platform strength (shipped & green):**
- 31 contract-routed server modules, 106 typed contracts, 865 integration + 282 unit tests green.
- Full role system (student / teacher / admin / parent), messaging + sockets, notifications + FCM, gamification, certificates + ijazahs with public verify, halaqa rooms, admin analytics, audit log, exports, load-tested (p95 authed read 6ms).
- **Shipped since the 07-14 draft:** real KFGQPC Madani Mushaf image reader (604 pages, pixel-faithful, swipe + zoom); verified ayah dataset (6,236 rows with page/juz mapping); **one-teacher-per-student enforced** end-to-end (server 403 + locked booking UI + admin change-request flow reachable from every student account); dropdown date/time pickers on booking.
- Parent **weekly digest already exists** server-side (`digest.service.ts`, tested). Offline query cache **already wired** (`PersistQueryClientProvider` in `_layout.tsx`).

**The honest gaps (what a Quran academy actually pays for):**

| # | Gap | Evidence |
|---|-----|----------|
| G1 | **The core hifz loop is broken in half.** The reader shows the real Mushaf, but tapping-to-log memorization was lost in the image switch; `MemorizationProgress` is surah-count-based and now has no input path from the reader. | `mushaf.tsx` has no memorization action; `logAyahMemorization` is orphaned. |
| G2 | **Recordings are disconnected from the Quran.** A recording is just a file + name; a teacher reviews audio with no page/surah context on screen. | `Recording` model has no surah/page/ayah fields. |
| G3 | **Revision scheduling is manual.** `revision.service.ts` is teacher-CRUD; `weak-ayahs` flags exist but feed nothing. No Sabaq/Sabqi/Manzil engine. | No scheduler job; revisions created one-by-one. |
| G4 | **Zero onboarding.** New users land on an empty home. (S2 of the 07-14 draft, never built.) | `app/onboarding/` doesn't exist; no `onboardingCompletedAt`. |
| G5 | **No acquisition surface.** Verify page is a bare fact page; nothing shareable. (S3, never built.) | No `public/` module, no `AcademyProfile`, no share image. |
| G6 | **No academy-level view.** Admin analytics exist per-metric, but no one-pager a director shows a board. (S5, never built.) | No `/admin/academy-health`. |
| G7 | **Deploy blockers.** (a) No migration creates the `surahs` table — fresh `migrate reset` fails (workaround was `db push`); (b) the 604 Mushaf images (~51MB) exist only on this machine (git-ignored, no distribution story). | Documented 2026-07-13; `extract_mushaf_pages.py` is manual. |

**The 10x thesis:** the platform is a *solid management tool* (booking, grading, messaging — all commodity). The 10x value is making it the **daily hifz instrument**: the student's memorization lives on the real Mushaf page, every recitation is anchored to that page, and revision runs itself. That loop is what no generic tutoring app has, what parents pay for, and what compounds daily usage. Everything else (onboarding, sharing, academy health) multiplies a loop that must exist first.

### H1 — The Hifz Engine

#### F1 · Page-level memorization on the real Mushaf  (2 days)

Hifz is memorized **by page** (the 604-page Madani layout is the unit every hafiz and teacher already uses — "I know 3 juz and 4 pages"). Replace the orphaned per-ayah model with page-level tracking driven from the reader.

**Build:** `PageMemorization { userId, page (1–604), status NOT_STARTED|LEARNING|MEMORIZED|SOLID, lastReviewedAt, updatedAt, @@unique([userId,page]) }` + contracts `getMyPages`, `setPageStatus`. Reader gains a status chip on each page (mark Learning / Memorized in ≤2 taps, student-only). Progress everywhere (home, memorization screen, parent view, teacher roster) derives from `memorizedPages/604` and juz completion — surah-level `MemorizationProgress` stays for grading but stops being the headline number.

**Acceptance criteria:**
- **AC1.1** From any Mushaf page, a student marks it Learning/Memorized in ≤ 2 taps without leaving the reader; the chip reflects status on revisit and after app restart.
- **AC1.2** `GET /mushaf/my-pages` returns all statuses in one call (≤ 300ms for 604 rows); the reader paints chips from a single fetch, not per-page requests.
- **AC1.3** Student home + memorization screen show pages-memorized, %, and per-juz breakdown computed from `PageMemorization`; teacher `student-detail` and parent child view show the same number (one source of truth).
- **AC1.4** Only the student (or their assigned teacher) can write their page statuses; cross-student writes → 403 (authz-matrix covered).
- **AC1.5** Migration is additive; `prisma migrate reset` succeeds on a fresh DB (depends on F4a landing first or together).
- **AC1.6** Marking a page MEMORIZED records `lastReviewedAt` and feeds F3's queue.

#### F2 · Recite-from-the-page  (1.5 days)

Anchor every recording to the Quran. Record directly from the Mushaf page; the teacher reviews with the exact page image beside the audio.

**Build:** additive `Recording.page Int?` (+ optional `surahId`); mic button on the reader → existing `expo-av` recorder pre-tagged with the current page; teacher review screen renders the page image (already served at `/mushaf-pages/<n>.webp`) alongside playback; weak-ayah flagging becomes one tap on that review screen (flags feed F3).

**Acceptance criteria:**
- **AC2.1** From a Mushaf page, start recording in ≤ 2 taps; upload carries `page`; no parallel recording code path (reuses the existing uploader + guards).
- **AC2.2** Teacher review of a page-tagged recording shows the page image + audio on one screen; untagged legacy recordings render exactly as today (nullable column, zero regression — pinned by existing media-flow tests staying green).
- **AC2.3** Teacher can flag a weak ayah from the review screen in ≤ 2 taps; flag lands in `WeakAyahFlag` for that student.
- **AC2.4** Recording list rows show the page/surah tag; tapping it opens the reader at that page.
- **AC2.5** Upload still enforces the assigned-teacher relationship guard; contract + itests updated additively (existing 865 stay green).

#### F3 · Self-running revision queue (Sabaq · Sabqi · Manzil)  (2 days)

The classical review ladder, automated: new material (sabaq) reviews daily, recent pages (sabqi) every 2–3 days, old pages (manzil) weekly — computed, not hand-scheduled.

**Build:** a pure function `buildRevisionQueue(userId, date)` over `PageMemorization` (recency-banded intervals) + `WeakAyahFlag` (weak pages surface more often) + existing `RevisionSchedule` (teacher overrides always win). Student home gets a "Today's revision" card: ordered pages, each deep-linking into the reader; completing one stamps `lastReviewedAt`. Nightly BullMQ job (graceful no-Redis fallback = compute on read) + optional reminder push through the existing notification service.

**Acceptance criteria:**
- **AC3.1** A student with ≥ 1 memorized page always has a non-empty, deterministic daily queue: same inputs → same queue (unit-tested pure function; no RNG).
- **AC3.2** Queue order: overdue manzil first, then sabqi, then sabaq; pages containing an active weak-ayah flag are boosted one band (each rule unit-tested).
- **AC3.3** "Reviewed" from the reader/card updates `lastReviewedAt` and removes the page from today's queue without a refetch (optimistic update).
- **AC3.4** Teacher-created `RevisionSchedule` rows appear at the top of the student's queue and are never dropped by the algorithm.
- **AC3.5** Works with Redis absent (computed on request) and with Redis present (nightly precompute + cached read ≤ 100ms) — both paths itested.
- **AC3.6** Parent child-view and teacher roster show revision adherence (done/queued this week) from the same data.

#### F4 · Deploy unblockers  (1 day, parallel)

**(a) Migration baseline.** Repair the ledger so a fresh database builds from migrations alone.
- **AC4.1** `npx prisma migrate reset --force` on an empty Postgres completes with zero manual steps (the `surahs`-table gap is captured in a baseline migration); CI job proves it on every PR.
- **AC4.2** `db push` is no longer required anywhere in docs or scripts; CLAUDE.md updated.

**(b) Mushaf asset pipeline.** The 604 page images become a reproducible artifact, not a laptop-local folder.
- **AC4.3** A documented one-command path populates `mushaf-pages/` on any machine (re-run `extract_mushaf_pages.py` against the source PDF, or download a checksummed archive from object storage — decision at plan time).
- **AC4.4** Server refuses to start in production mode with an empty/partial `mushaf-pages/` unless explicitly overridden (env flag) — fail loud, not 404s at runtime.
- **AC4.5** `GET /mushaf-pages/1.webp` and `/604.webp` return 200 + correct checksums in a smoke itest.

### H2 — Activation & Teacher Leverage

#### F5 · Role onboarding wizards  (2 days) — carried from S2, updated

As specified in the 07-14 draft §4-S2 (schema `User.onboardingCompletedAt`, `POST /account/complete-onboarding`, three role flows), with two updates forced by this week's changes:
- Student step 2 shows **the one assigned teacher** (now guaranteed by the enforcement work) or the "request a teacher from admin" prompt that already exists on the booking form — same component, no new flow.
- Student step 3 ("record your first surah") records **from the Mushaf page** via F2, not a bare recorder.

**Acceptance criteria:** AC2.1.1–AC2.3.5 from the 07-14 spec hold verbatim, plus:
- **AC5.1** Student wizard's teacher step reuses the assigned-teacher derivation from `appointments.tsx` (profile → ACCEPTED-appointment fallback); no third derivation is introduced.
- **AC5.2** Completing the student wizard leaves the student with: teacher greeted (message sent), ≥ 1 page-anchored recording, and the F3 revision card visible on home.

#### F6 · Teacher cockpit rathink (7 screens)  (2 days) — carried from S1

AC1–AC7 from the 07-14 spec hold verbatim (daily landing surfaces now-actions; grade in ≤ 3 taps; at-risk reasons; one-tab plan creator; sorted review queue; full a11y/i18n/theme gates; impeccable-critique improvement), plus:
- **AC6.1** The review queue and student detail integrate F2's page-anchored review (page image beside audio) — the rathink designs around it, not the old audio-only card.
- **AC6.2** Roster rows show page-based progress and revision adherence from F1/F3.

#### F7 · Streak-risk nudges + digest surfacing  (1 day)

The digest and notification infra exist; connect them to the loop.
- **AC7.1** A student whose streak lapses at end-of-day-minus-2h with no activity receives one push (never more than one/day; opt-out respected via existing notification prefs; unit-tested window logic).
- **AC7.2** The parent weekly digest gains pages-memorized-this-week + revision adherence (from F1/F3); existing digest tests extended, not replaced.
- **AC7.3** Digest send remains idempotent per week per parent (existing pin holds).

### H3 — Acquisition & Academy-Readiness

#### F8 · Public landing + certificate share image  (1 day) — carried from S3

Build exactly as specified in the 07-14 spec §3.3/§4-S3: `AcademyProfile`, `public/` module, extended verify, 1200×630 share PNG (≤ 200KB, revoked → 404, no PII beyond current verify page), public mobile route group. **AC3.1–AC3.7 hold verbatim.** One addition:
- **AC8.1** The share sheet is WhatsApp-first in ordering/copy (the dominant channel for the target audience); share URL opens the improved verify page with academy branding.

#### F9 · Academy Health one-pager  (0.5 day) — carried from S5 (report half)

Build the `GET /admin/academy-health` aggregate + admin screen exactly per the 07-14 spec §3.4, with the response extended by two loop metrics: `learning.pagesMemorizedThisWeek` and `learning.revisionAdherencePct` (from F1/F3). Original ACs hold (≤ 2s read, Redis-cached with graceful fallback, PDF export ≤ 5s, cache asserted by itest).

#### F10 · Admin + Parent + Shared UX rathink (12 screens)  (1 day) — carried from S5 (UX half)

AC5.1–AC5.5 from the 07-14 spec hold verbatim (admin one-pager, filterable audit log, parent home as child-summary cards, notification→conversation deep-links, full gates). De-scope fallback unchanged: if tight, ship admin home + parent home, defer shared polish.

#### F11 · Offline resilience  (1 day)

The infra is half-there (query persistence wired, `expo-image` disk cache on). Finish it for the contexts this app lives in (mosques/madrasas with weak connectivity):
- **AC11.1** "Download my Mushaf" action prefetches the student's memorized + queued pages (bounded set, not all 604) to the image disk cache with a progress bar; those pages open in airplane mode.
- **AC11.2** Page-status marks (F1) and revision completions (F3) made offline queue and sync on reconnect (TanStack mutation persistence — already dehydrating mutations in `_layout.tsx`); a visible "pending sync" badge shows queued count.
- **AC11.3** Recording upload retries automatically on reconnect (bounded retries, user-visible failure state); no silent loss — a queued-then-failed upload is recoverable from the recordings screen.
- **AC11.4** Read-only screens (home, grades, mushaf-cached pages) render last-known data offline instead of error states; no infinite spinners (each screen spot-checked in airplane mode).

**Risks (delta from stage-2 §7, which still applies to carried items):**

| Risk | Mitigation |
|---|---|
| F1 page-status writes race with F3 queue reads | Queue is a pure function over rows; no denormalized counters to corrupt |
| F3 interval tuning is pedagogy, not code | Ship conservative classical defaults (1/3/7-day bands); teacher override (AC3.4) is the escape hatch; make bands config constants, not schema |
| F11 offline sync conflicts (page marked on two devices) | Last-write-wins on `updatedAt` — acceptable for single-student-owned rows; documented |
| Carried UX rathinks (F6, F10) still need their brainstorm gate | Unchanged process: brainstorm → spec → plan per cluster before any screen code |

## 5. Definition of done (roadmap-level)

1. A student's daily loop works end-to-end offline-tolerant: open app → today's revision queue → read the real Mushaf page → recite & record from it → teacher reviews with the page on screen → progress and streak update — with zero manual scheduling by anyone.
2. Pages-memorized is the single progress number shown to student, teacher, parent, and the academy-health report.
3. A new user of any role reaches their first meaningful action in ≤ 90 seconds (F5, measured).
4. An academy director can open one screen (F9) and share one certificate image (F8) to pitch the platform.
5. A fresh clone + empty database reaches a running, fully-populated system (migrations, seed, ayahs, mushaf images) with documented one-command steps (F4).
6. All existing suites stay green throughout; every new endpoint is in the authz matrix; `security-reviewer` signs off on F5 (auth flow), F8 (public), F9 (admin).
